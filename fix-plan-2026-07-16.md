## Phase 2 — Reference numbering migration (O1, O2) + types regen (B9) · ~1–1.5 days

This is the Phase-0 debt the July remediation deferred and never wrote. Order matters: dedup → indexes → generator.

**2.1 · Dedup existing collided references.**
New `supabase/reference-renumber-2026-07.sql`: for each of the 9 tables (entry_records, object_exits, loans, rights_records, collection_use_records, valuations, damage_reports, collections_reviews, disposal_records), renumber museum-wide per prefix-year ordered by `created_at` (window function `row_number() OVER (PARTITION BY museum_id, year ORDER BY created_at)`), rewriting only rows whose ref collides. Keep the existing `XX-YYYY-NNN` format. Run against live via SQL editor / MCP, commit the file.

**2.2 · Add the UNIQUE indexes.**
Same migration file: `CREATE UNIQUE INDEX IF NOT EXISTS ... ON <table> (museum_id, <ref_column>)` for all 9. (Live DB verified to have none as of 16 July.)

**2.3 · Shared museum-wide reference generator.**
New `lib/nextReference.ts`: `nextReference(supabase, table, column, prefix, museumId)` → `SELECT count` museum-wide for the current year → `PREFIX-YYYY-{n+1}`, plus insert-retry-on-23505 (bump and retry up to 3×).
Switch the 7 per-object generators to it: `ExitsTab.tsx:122`, `LoansTab.tsx:90`, `DamageTab.tsx:132`, `ValuationTab.tsx:65`, `RightsTab.tsx:94`, `ConditionTab.tsx:86`, `ConservationTab.tsx:204`. (Register pages — entry/use/review/disposal — already count museum-wide; move them onto the helper too so retry logic is shared.)

**2.4 · B9 regenerate `lib/database.types.ts`** from the live project (picks up `object_exits.returned_date/returned_condition` and anything else stale). Do it in this phase because 2.1/2.2 also touch the DB.

## Phase 3 — Row cap & pagination (O3) · ~1–2 days

Confirmed live: no `db.max_rows` override → PostgREST caps every query at 1,000 rows. At >1,000 objects, compliance scores, registers, and exports silently truncate.

**3.1 · Counts, not row-fetches.** `app/dashboard/docs/page.tsx:151-172`: replace the 20 `select('*')` fetches used only for numerators/denominators with `select('id', { count: 'exact', head: true })` (or targeted `select(col)` + `count`). Compliance math then never depends on row limits.
**3.2 · Paginate the objects list.** `app/dashboard/page.tsx`: `.range()`-based paging (50–100/page) + `count: 'exact'` for the total; search/filter server-side via existing `.ilike`.
**3.3 · Chunked export.** `app/api/export/objects/route.ts` (and the sidebar JSON export): loop `.range(offset, offset+999)` until short page; stream/accumulate.
**3.4 · Register pages** (locations, loans, exits, insurance, damage, risk, rights, reproductions, use, review, disposal, audit): same `.range()` pattern where they currently fetch unbounded. Can be rolled out incrementally — objects/docs/export first (3.1–3.3 are the correctness-critical ones).
**3.5 ·** Do **not** raise `max_rows` as the fix — it just moves the cliff. (Optionally raise to 5,000 as a stopgap the day a big museum signs, but 3.1–3.3 are the real fix.)
